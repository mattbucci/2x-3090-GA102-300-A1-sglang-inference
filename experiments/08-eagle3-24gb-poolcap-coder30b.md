## Objective

The coder-30b EAGLE3 lane delivers 1.65x decode but is hard-capped at CTX=16K, which 97.3% of SWE-bench agentic prompts exceed (median 41K) — so the win never touches the mission workload. This experiment tests the repo's longest-standing recorded-but-untested idea: cap the KV pool explicitly with --max-total-tokens (instead of starving everything via MEM=0.70) so the draft + its cuda graphs fit at CTX=32-96K on 2x24GB, and use --speculative-draft-window-size to blunt the known deep-KV draft re-attention collapse. A clean null closes the idea permanently; a win produces the sizing data for the dynamic spec-to-no-spec fallback (spec plan task #17) that the 256K-only production constraint requires before any promotion.


## Hypothesis

With the KV pool capped to CTX+margin via --max-total-tokens (freeing the VRAM the auto-profiler would otherwise claim), coder-30b native-AWQ + EAGLE3 (steps 4 / topk 4 / draft 8) boots WITHOUT OOM at CTX=65536 (stretch: 98304) on 2x24GB, and delivers >=1.2x server-log decode speedup over no-spec at 41-64K true depth on real-code NOVEL-generation content with accept_len >= 2; adding --speculative-draft-window-size 8192 holds decode tok/s at depth without dropping accept_len below 2. Falsified by: OOM at CTX=32768 in every MEM/cap combination tried, or spec &lt;= no-spec at 41K+ depth.

## Method

1. Preflight on the 3090 box: confirm no bake-off/scoring/calibration is active (sudo docker ps; systemctl status swebench-bakeoff) — Rule 1/Rule 2 forbid concurrent GPU/docker workloads; note this lever in the README before starting per the iteration-loop discipline.
2. Verify assets and record exact on-box paths (they cannot be confirmed from the R9700 checkout; the 2026-05-29 receipt used /data/models/* while launch.sh now uses $MODELS_DIR=$HOME/AI/models): (a) draft dir $MODELS_DIR/drafts/eagle3-coder30b exists and its config.json architectures == ['LlamaForCausalLMEagle3'] (if missing: hf download lmsys/SGLang-EAGLE3-Qwen3-Coder-30B-A3B-Instruct-SpecForge, 361 MB); (b) a native-AWQ (quant_method=awq) Qwen3-Coder-30B-A3B target dir exists — the preset default $MODELS_DIR/Qwen3-Coder-30B-A3B-AWQ-CT is CT-format and NOT valid for QUANT=awq_marlin; if the native-AWQ dir is gone, the experiment needs a CT->native conversion or HF mirror pull first (hours, not blocking-forever).
3. Build the instrument scripts/specforge/perf_coder30b_poolcap.py as a SERVER-CLIENT (NOT the in-process delta engine): drive the running launch.sh server over HTTP (copyheavy-style requests), read the SERVER-LOG 'gen throughput' as the AUTHORITATIVE decode tok/s (client/delta numbers are correlation-only and under-measure bursty spec ~2x), fetch accept_len from /get_server_info (new call — no donor has it), use the NOVEL-code prompts from perf_devstral_spec.py (Dijkstra / merge-intervals set), and point any source-padding glob at an on-box tree that EXISTS — /data/sglang-rebase-v0515 (per common.sh:37), NOT copyheavy's hardcoded stale /data/sglang-rebase-v0512. Discard perf_devstral's sgl.Engine mechanism entirely.
4. Arm B — re-validate the status-quo 16K lane on v0.5.15 (mandatory gate; SGLANG_ENABLE_SPEC_V2 was removed and the lane was last proven on v0.5.12): SPEC_DECODE=1 QUANT=awq_marlin MODEL=<native-awq-dir> ./scripts/launch.sh coder-30b — gate: boots, python scripts/eval/probe_codegen.py --port $PORT output coherent (probe_codegen defaults to port 30000 but the stack serves PORT=23334 per common.sh:39, so --port $PORT is required or the probe hits a dead port), accept_len >= 2 from /get_server_info, tok/s within ~20% of the 306 tok/s receipt. If it fails to boot, STOP: that is a standalone v0.5.15-regression finding (receipt it); the pool-cap experiment is BLOCKED, not failed.
5. Arm A — no-spec baseline at true depth: production coder-30b preset (no SPEC_DECODE, same MODEL/QUANT), measure decode tok/s at ~16K/~41K/~64K/~96K depth on real-code NOVEL content via perf_coder30b_poolcap.py; authoritative number = server-log gen throughput; log nvidia-smi per-card VRAM. This is the fixed depth ladder every spec arm's ratio is computed against.
6. Arm C — isolate the pool-cap mechanism for NO-REGRESSION at unchanged CTX (one mechanism at a time): SPEC_DECODE=1 QUANT=awq_marlin MODEL=<native-awq-dir> OVERRIDE_ARGS='--max-total-tokens 18432' ./scripts/launch.sh coder-30b --mem-fraction 0.80 — gate: boots, tok/s PARITY with Arm B at 16K (this parity check IS the valid isolation). Do NOT treat Arm C's VRAM as the reclaim number: at 16K the naturally-profiled pool already ~= the cap so there is ~zero reclaim to see, and Arm C also moves MEM 0.70->0.80, so its VRAM delta is not a clean single lever. The reclaim measurement lives in Arm D.
7. Arm D — CTX ladder with pool cap, sweeping the FULL Arm-A depth ladder per bootable server: for CTX in 32768, 65536, 98304 run SPEC_DECODE=1 QUANT=awq_marlin MODEL=<native-awq-dir> OVERRIDE_ARGS="--max-total-tokens $((CTX+2048))" ./scripts/launch.sh coder-30b --context-length $CTX --mem-fraction 0.80 (retry an OOM rung once at --mem-fraction 0.75 before declaring it dead). At each bootable rung, measure spec decode tok/s at EVERY Arm-A depth <= (CTX-4K) on real-code NOVEL content — e.g. the 65536 server yields BOTH the 41K and ~61K(~64K) points on one boot, the 98304 server adds ~94K — plus accept_len from /get_server_info and per-card VRAM. RECLAIM MEASUREMENT: at one high-CTX rung capture per-card VRAM with the cap ON vs a cap-OFF boot attempt at the same CTX/MEM (the cap-off OOM, or its higher VRAM if it boots, IS the reclaim finding on this ~1x-provisioned pool).
8. Arm E — isolate the window mechanism at the deepest bootable rung: rerun that rung with OVERRIDE_ARGS+=' --speculative-draft-window-size 8192' (must be >= 8, the num-draft-tokens floor; optionally also 2048) and A/B tok/s + accept_len against Arm D at the SAME depth(s) — this directly probes the deep-KV re-attention collapse vs acceptance tradeoff.
9. Publish the receipt: benchmarks/quality/coder30b-eagle3-poolcap-2026-07-<dd>.json in the devstral-eagle3-speedup.json shape (per-depth nospec/spec/speedup/accept_len, per-arm VRAM, exact launch commands, engine=v0.5.15) + the new perf script; update README.md:13 fleet-audit checkbox and the README.md:80 Decode-ideas bullet with the WIN/NULL one-liner; commit at a clean boundary.
10. If WIN: do NOT change preset defaults or claim production promotion (user constraint: only 256K servers ship; promotion is gated on the dynamic spec->no-spec fallback, spec plan task #17) — instead append the measured band/VRAM numbers to evals/swebench/spec_decode_plan.md as task #17 sizing data, and push the portable window-flag finding to R9700's README (they serve EAGLE3 with 32 GB headroom).
11. Any server run or sweep expected >30 min: detach via the setsid pattern from CLAUDE.md with logs to a kept file (the server log IS the measurement instrument — always redirect it to a kept path).

## Success criteria

- Memory result: coder-30b + EAGLE3 boots and completes a full depth sweep without OOM at CTX >= 65536 with the pool cap (receipt: launch log + nvidia-smi capture per arm in the receipt JSON).
- Perf result: spec/no-spec server-log gen-throughput ratio >= 1.2x at ~41K AND ~64K true depth on real-code NOVEL generation, with accept_len >= 2 (/get_server_info) — both points measured on the SPEC side by perf_coder30b_poolcap.py against the matching Arm A depth (the 65536 rung supplies both 41K and ~64K on one boot per the Arm D ladder), not a single depth per rung.
- Isolation: Arm C shows pool-cap causes no tok/s regression at 16K (parity vs Arm B), and Arm E reports the window's independent effect on tok/s + accept_len at the deepest rung (both A/B'd, one mechanism per arm); the pool-cap reclaim number is reported from the high-CTX cap-on vs cap-off comparison in Arm D, not from the 16K arm.
- Receipt hygiene: benchmarks/quality/coder30b-eagle3-poolcap-2026-07-<dd>.json committed with exact commands + engine version; README audit checkbox and Decode-ideas bullet updated; no preset default flipped.
- A NULL meeting the kill criteria below, fully receipted, also counts as success — it closes the repo's longest-standing open idea (the audit bullet's stated value).


## Kill criteria

- Arm B fails to boot or is incoherent on v0.5.15 and the fix is not obvious within ~2 h: stop, record the v0.5.15 spec-lane regression as its own finding; the experiment is BLOCKED (separate item), not null.
- Memory null: CTX=32768 OOMs with the pool cap at both MEM=0.80 and MEM=0.75 (and the Arm D high-CTX cap-on vs cap-off VRAM comparison confirms ~zero reclaim on the ~1x-provisioned pool): record NULL 'pool-cap frees nothing on pure-attention coder-30b', close the README bullet.
- Depth null: every bootable rung shows spec <= no-spec at >= 41K real-content NOVEL depth (acceptance collapse) even with the window: record NULL 'EAGLE3 band does not extend past ~16-32K on 24 GB regardless of memory', close the bullet; NGRAM remains the only depth spec.
- Budget guard: > ~10 server boots or > 1 GPU-day consumed without a bootable CTX >= 32768 spec config: stop and record the strongest partial result as the null.
- If a kernel-BUG reboot bisects a sweep, resume rather than re-plan; if reboots make any single arm unrepeatable twice, park with receipts and note the blocker.

## Constraints

- READ-ONLY authoring note: this spec was written from the R9700 checkout; all execution happens on the 3090 box by its team.
- Rule 1 / Rule 2: no concurrent calibration+serving/eval and no concurrent rollout+score; only one GPU/docker workload at a time on the eval box.
- Measurement invariants: decode tok/s from server-log gen throughput at TRUE depth on real NOVEL content (client TPOT under-measures bursty spec ~2x; unpinned random lengths lie — pin --random-range-ratio 1 if bench_serving is used); one mechanism changed per arm with A/B at short+medium+deep; the SPEC-side depth ladder must match the Arm-A baseline ladder so ratios are directly computable.
- Production promotion is out of scope: user constraint = only 256K production servers; a 64-96K spec server can only ship behind the task-#17 dynamic fallback. Do not flip preset defaults.

- Negative results are findings: every null goes in the receipt JSON + README one-liner, not a silent drop.


## Risks

- v0.5.15 spec-V2-only worker may break the lane that was validated on v0.5.12 (SGLANG_ENABLE_SPEC_V2 removed/ignored; overlap scheduler default changed semantics) — mitigated by the Arm B gate with an explicit 2h stop-loss.
- The audit's ~1x-provisioning caveat may mean --max-total-tokens reclaims almost nothing on coder-30b (the 3-9x over-provisioning is qwen36-hybrid-specific) — the ~5 GB draft-cuda-graph cost would then still not fit; this is the primary null path and is cheap to hit (Arm D's high-CTX cap-on/cap-off comparison measures it directly).
- Draft acceptance may collapse before 41K (the draft trained short-context; README already documents depth collapse at 256K) — the window flag is the counter-lever being tested, but it can also cut accept_len below 2 (Arm E measures the tradeoff).
- Kernel-BUG reboot (~9-17h under I/O) can bisect a sweep — keep arms short, logs on disk, and resume per CLAUDE.md forensics.
- Path drift: the 2026-05-29 receipt used /data/models/* while launch.sh now uses $MODELS_DIR=$HOME/AI/models, and the copyheavy donor hardcodes the stale v0512 padding tree — step 2 resolves actual on-box model paths and step 3 repoints the padding glob at /data/sglang-rebase-v0515 before any launch; if the native-AWQ target dir no longer exists on the 3090 box, the experiment needs the CT->native conversion or an HF mirror pull first (hours, not blocking-forever).
- --num-continuous-decode-steps 8 (preset DECODE_STEPS) interacting with the spec worker is untested on v0.5.15; if Arm B misbehaves, A/B --decode-steps 4 before declaring the lane broken.
- Instrument-build risk: the two donors have different execution models (perf_devstral is in-process sgl.Engine delta; copyheavy is an HTTP client) and copyheavy also builds a copy-heavy prompt and calls no /get_server_info — building the server-client cleanly (novel prompts, server-log authoritative, new accept_len fetch, correct port 23334) is real new-script work, not a graft.
