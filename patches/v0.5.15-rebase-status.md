**NEW patch (caught by validation — the one functional break the transformers bump introduced):**
- **057 mistral-common-backend-optout** — transformers 5.12.1 routes Mistral-family checkpoints
  that ship a `tekken.json` to the new **`MistralCommonBackend`** tokenizer, which by design never
  parses special tokens out of text. SGLang renders the chat jinja to a STRING and then encodes
  it, so `[INST]` / `[/INST]` / `[TOOL_CALLS]` entered devstral prompts as PLAIN TEXT (`<s>[INST]`
  → pieces `['<','s','>[','IN','ST',']']` instead of `[1, 3]`): first sweep pass measured needle
  **0.0 at every depth incl. 1K**, HumanEval 80%→48%, tool_call capability dead, and generation
  blowing past turn ends into hallucinated `[INST]` turns — while `basic` still "passed" (the
  answer text appears before the ramble), so only a quality probe catches it. Fix: in
  `get_tokenizer`, when AutoTokenizer returns `MistralCommonBackend`, reload with
  `fix_mistral_regex=False` — the kwarg's *presence* is transformers' documented escape route to
  `TokenizersBackend` (`"fix_mistral_regex" not in kwargs` gate in `tokenization_auto.py`), and
  `False` loads `tokenizer.json` verbatim. Token ids verified **bit-identical to tx 5.8.1**
  (`[INST]`→3, `[TOOL_CALLS]`→9); the rerouted result must also skip sglang's
  `_resolve_tokenizers_backend` re-resolution (it would take the MistralCommon route again and
  crash on that backend's strict kwarg validation). AutoProcessor's inner tokenizer (Pixtral
  vision path) already takes the TokenizersBackend route — text-tokenizer-only fix.
  **Cross-team relevance: any stack on the tx 5.12.x pin serving Mistral-family ships hits this**
  (R9700 FP8 devstral included).
