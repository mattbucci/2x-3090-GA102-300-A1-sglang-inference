# AWQ scales audit — 2026-05-07 sweep

Ran `scripts/eval/check_awq_scales.py` (post-bf16 fix `6cfcd21`) across every
local AWQ checkpoint + every `hf-mattbucci/` mirror. 30+ dirs scanned;
findings below distinguish disaster-class (>=100 flagged scale tensors —
recipe-side calibration broken) from minor-flag (≤5 flagged — classic
rare-expert under-calibration in a single early-layer expert).

Pattern: ALL-ZERO scales in vast portions of vision_tower + experts. This
is the v3-disaster pattern — empty `ignore` regex during calibration
silently skipped quantization for descendants like
`embed_vision.embedding_projection`, producing zero scales that
dequantize all-zero at inference. **Fix path:** full re-calibration with
proper `ignore` regex (`re:.*vision_tower.*` + `re:.*embed_vision.*` +
`re:.*multi_modal_projector.*`).

Pattern: 52-58% zero scales on specific `experts.{N}.gate_proj.scales` +
`experts.{N}.up_proj.scales` pairs in **layer 0** (Qwen3.6 family) or
**layer 1** (Qwen3.5 family). Classic MoE GPTQ rare-expert
under-calibration: most tokens never route to those experts during the
calibration corpus, so their gate/up projections never see Hessian
updates and end up with degenerate scales.

**Inference impact:** when the router happens to pick a flagged expert
(routing distribution-dependent — typically <5% of tokens), that expert
produces degraded output. The other 95%+ of tokens are unaffected. Not a
serving-blocker but a quality-degradation footgun on tasks that
stress-test rare-token paths (long-tail vocabulary, code edge cases,
specific domain jargon).

**Mitigation paths for next recal cycle:**
- Bump `NUM_CALIBRATION_SAMPLES` 512→1024+ so rare experts see more activations
- Port the Gemma-4-style `force_route_all_experts` monkey-patch to Qwen3.5 / 3.6
- Add `check_awq_scales.py` to `run_full_pipeline.sh` post-conversion gate
  (already done for `convert_moe_ct_to_awq.py` in `4f57767`)

## Key insight: Qwen3.6-35B native vs CT divergence

The same model `Qwen3.6-35B-A3B` exists in two builds:
- `hf-mattbucci/Qwen3.6-35B-A3B-AWQ` (native AWQ, 20G, **144 flagged**)
- `hf-mattbucci/Qwen3.6-35B-A3B-AWQ-CT` (CT format, 19G, **clean**)

Different file sizes + different scale-tensor counts (30970 vs 31010)
indicate these are **two different calibration runs**, not just format
conversions of the same source. The CT build is calibration-clean.

**Production `qwen36` preset currently points at the flagged native
build.** Consider validating the CT variant via launch + capability
sweep, then switching the preset if it serves correctly. Caveat:
SGLang's NVIDIA CT loader has a known issue with `(1, H)` shaped
`shared_expert_gate` (per `Known Issues` in main README), so the CT
variant may not load on NVIDIA without a workaround.

## Audit script known limitation

5 BF16 base model directories (`*-BF16`, `gemma-4-26B-A4B-it-BF16`,
etc.) returned `rc=1` despite having 0 flagged scales — they have no
`*.scales` tensors at all (not quantized). The audit script returns
`rc=1` for any "no scales found" case rather than just for genuine
flags. Worth a follow-up to make the script return `rc=0` when 0 scales
are scanned (since "not an AWQ build" is not a failure mode); not
blocking for this iteration.
