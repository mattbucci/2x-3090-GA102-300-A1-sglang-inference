Pattern: 52-58% zero scales on specific `experts.{N}.gate_proj.scales` +
`experts.{N}.up_proj.scales` pairs in **layer 0** (Qwen3.6 family) or
**layer 1** (Qwen3.5 family). Classic MoE GPTQ rare-expert
under-calibration: most tokens never route to those experts during the
calibration corpus, so their gate/up projections never see Hessian
updates and end up with degenerate scales.

**Inference impact:** when the router happens to pick a flagged expert
(routing distribution-dependent — typically <5% of tokens), that expert
produces degraded output. The other 95%+ of tokens are unaffected. Not a
serving-blocker but a quality-degradation footgun on tasks that
stress-test rare-token paths (long-tail vocabulary, code edge cases,
specific domain jargon).

**Mitigation paths for next recal cycle:**
- Bump `NUM_CALIBRATION_SAMPLES` 512→1024+ so rare experts see more activations
- Port the Gemma-4-style `force_route_all_experts` monkey-patch to Qwen3.5 / 3.6
- Add `check_awq_scales.py` to `run_full_pipeline.sh` post-conversion gate
  (already done for `convert_moe_ct_to_awq.py` in `4f57767`)

## Key insight: Qwen3.6-35B native vs CT divergence

The same model `Qwen3.6-35B-A3B` exists in two builds:
- `hf-mattbucci/Qwen3.6-35B-A3B-AWQ` (native AWQ, 20G, **144 flagged**)
- `hf-mattbucci/Qwen3.6-35B-A3B-AWQ-CT` (CT format, 19G, **clean**)

Different file sizes + different scale-tensor counts (30970 vs 31010)
indicate these are **two different calibration runs**, not just format
conversions of the same source. The CT build is calibration-clean.

**Production `qwen36` preset currently points at the flagged native
build.** Consider validating the CT variant via launch + capability
sweep, then switching the preset if it serves correctly.

**2026-05-07 update — CT variant CONFIRMED BROKEN on NVIDIA.** Launched
`hf-mattbucci/Qwen3.6-35B-A3B-AWQ-CT` via `qwen36` preset at TP=1 / 2K /
quant=compressed-tensors. Server boots cleanly via
`CompressedTensorsWNA16MarlinMoEMethod`, but logs **120 warnings** of
the form:

  `Parameter model.layers.{0..39}.mlp.shared_expert_gate.weight_packed not found in params_dict`
  `... .weight_scale not found in params_dict`
  `... .weight_shape not found in params_dict`

Validator: 1/4 PASS (only video grep'd `red` from random gibberish).
Output is multilingual word-soup — model's effectively running with
random-init `shared_expert_gate` for all 40 layers.

**Forensic root cause (corrects earlier "name mapping" hypothesis):**
the CT safetensors *does* ship `weight_packed/scale/shape` for these
gates (and the load-side `model.language_model.X` → `model.X` rename at
`qwen3_5.py:1840` works correctly). The actual bug is that
**`shared_expert_gate` is constructed as plain `torch.nn.Linear` on the
GPU path** in `qwen2_moe.py:Qwen2MoeSparseMoeBlock.__init__` (Qwen3.5
inherits this), with no `quant_config` plumbed in. So the model's
params_dict has only `shared_expert_gate.weight`, never
`shared_expert_gate.weight_packed`. CT format ships the quantized
triplet which has nowhere to land.

**Native AWQ doesn't hit the bug** because the AWQ export script
silently skipped `shared_expert_gate` (probably because (H, 1) is
too narrow for group-quant) and exported it as BF16
`shared_expert_gate.weight` — which the plain nn.Linear loads fine.

**Two correct fix paths:**

1. **Recipe-side (R9700 should adopt):** add `re:.*shared_expert_gate.*`
   to the CT calibration `ignore` list. Then the CT export ships BF16
   `weight` like native AWQ does, and SGLang loads it cleanly. Does NOT
   require a serving-side patch. Same fix shape as the regex-ignore for
   `vision_tower` / `embed_vision` from the v3 disaster.

2. **Loader-side (3090 stack-side patch):** add on-the-fly
   dequantization in `qwen3_5.py:load_weights` for any `weight_packed`
   key whose target layer is a plain `nn.Linear`. Buffer the 3-tuple
   `(weight_packed, weight_scale, weight_shape)` per layer, dequant to
   BF16 once all 3 are seen, populate `weight`. More complex but works
   without a recal.

**Cross-team note shipped to R9700** at their commit `da00df8` (2026-05-07).
They own the recipe; option (1) is the cleaner fix for both stacks. Awaiting
R9700-side adoption decision.

## Audit script known limitation

5 BF16 base model directories (`*-BF16`, `gemma-4-26B-A4B-it-BF16`,
etc.) returned `rc=1` despite having 0 flagged scales — they have no
`*.scales` tensors at all (not quantized). The audit script returns
`rc=1` for any "no scales found" case rather than just for genuine
flags. Worth a follow-up to make the script return `rc=0` when 0 scales
are scanned (since "not an AWQ build" is not a failure mode); not
blocking for this iteration.
